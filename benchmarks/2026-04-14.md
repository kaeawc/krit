Track C is a pure code-motion change — it moved the 7 node-era
helpers from `internal/scanner/scanner.go` to

vary by ±100ms). Treat the four-run average of **~6428ms** as the

**After Track D** (164 archive functions deleted wholesale):
- Run 1: 6959ms, 5467 findings, 238 rules

Track D has no wall-time signal — expected, because the archive was
already unreachable code, so deleting it doesn't change what the
runtime does. Its wins are static: cleaner source tree, smaller
binary, and the cluster's invariant ("no `*sitter.Node` in production
rule code") is now enforceable by grep rather than by documentation.

Track F adds 12 new actively-dispatched rules (4 of 16 are marked
default-inactive). The +580ms vs Track-C reflects real work being
done: 12 new rules running on 2467 files produces 148 new findings
and adds ~48ms per rule, consistent with the existing per-rule
dispatch cost envelope. This is honest work, not overhead.

The aggregated `dispatchWalk` and `dispatchRuleCallbacks` counters
dropped by ~5.5s / ~4.2s respectively. These are per-file/per-rule
sums (not wall time), so they overcount under parallel execution, but
the drop is consistent with removing a large pocket of cgo-backed node
traversal from the rule execution hot path — which is the live-time
expression of the Track B work.

`typeIndex` is slightly up (+66ms). That's within run-to-run noise
(run 1 had it at 1818ms, run 2 at 1773ms — that's 45ms of noise on its
own). The story for the typeinfer subsystem under real workload is
"wash to small win" at the wall-time level; the bigger win is in the
aggregated dispatch counters, which should translate to less CPU time
spent across parallel rule execution on large codebases.

## Notes

- `dispatchWalk` and `dispatchRuleCallbacks` are aggregated
  per-file/per-rule counters from `internal/rules.RunStats`, not
  end-to-end wall time, so they exceed the total process duration
  under parallel execution. They're useful as "total work done" but
  not for apples-to-apples wall-time comparison.
- The post-closure run (after Track C + Track D) should be captured
  separately; at that point `scanner.File.Tree` drops out and the
  `flattenTree`-only path becomes the sole sitter-touching surface.
